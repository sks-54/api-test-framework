# /run-workflow

> **Before starting — enable auto-accept**
>
> This workflow runs 50–100 tool calls (file writes, edits, bash commands) in sequence.
> Without auto-accept, you will be prompted to approve every single one.
>
> - **CLI / terminal**: press `Shift+Tab` to cycle to auto-accept mode
> - **VS Code / JetBrains / Desktop**: click the permission mode indicator (bottom status bar) → select "Auto-accept edits"
>
> You can disable it again when the workflow finishes.
> Risky actions (git push, destructive operations) still prompt regardless of this setting.

Run the complete apitf workflow end-to-end from a spec file.

## What it does

Chains four automated steps without intervention:

1. **Parse** — extract endpoints from the spec (PDF, OpenAPI, Markdown)
2. **Scaffold** — generate validator + test suite. AI-generation is automatic when a provider
   is available (Claude Code CLI detected via `CLAUDECODE=1`, or `ANTHROPIC_API_KEY` set).
   Without either, produces a 5-test stub (no AI required).
3. **Eval loop** — run pytest, auto-fix structural errors (up to `--max-iter` rounds), categorise quality failures
4. **Reflector** — Opus reviews the final test file and scores it against the QA rubric (0–100, threshold 70)

## AI provider auto-discovery (controls stub vs. full generation)

`apitf-run` detects the best available provider automatically — no flag needed:

1. **Claude Code session** (`CLAUDECODE=1` env var) → `ClaudeCLIProvider`, zero config
2. **`ANTHROPIC_API_KEY`** set in environment or `.env` → `AnthropicProvider`
3. **Neither** → no AI; 5-test stub generated, reflector skipped

There is no CLI flag to toggle between stub and AI mode. To force stub-only, unset both
`CLAUDECODE` and `ANTHROPIC_API_KEY` before running.

## CLI command

```bash
# Default — AI generation if provider available, stub otherwise
apitf-run specs/myapi.pdf --env my_api --sample

# Override base URL (for multi-API PDFs where auto-detection picks the wrong host)
apitf-run specs/home_test.PDF --env countries \
  --base-url https://restcountries.com/v3.1 \
  --probe-path /name/germany \
  --sample

# Custom iteration limit and model overrides
apitf-run specs/petstore_spec.pdf --env my_api --sample \
  --max-iter 5 \
  --model claude-sonnet-4-6 \
  --reflector-model claude-opus-4-7

# Force sequential mode (disables ThreadPoolExecutor — useful for debugging)
apitf-run specs/petstore_spec.pdf --env my_api --sample --no-parallel
```

## Inside Claude Code (this session)

When asked to run the full workflow, execute:

```
apitf-run <spec_file> --env <env> --sample
```

Then:
1. Monitor the eval-loop output — each iteration is printed as it runs
2. If QUALITY failures remain after the loop, file GitHub issues and add `xfail` markers
3. Read the reflector score — if < 70, apply the listed corrections and re-run the eval step:
   ```
   python scripts/advisor_review.py --env <env> --test-file tests/test_<env>.py
   ```
4. Repeat until reflector score ≥ 70 or all deviations are filed as known bugs

## Run only the eval loop + reflector on an existing test file

```bash
python scripts/advisor_review.py \
  --env cf_petstore \
  --test-file tests/test_cf_petstore_full.py \
  --max-iter 3 \
  --reflector-model claude-opus-4-7
```

## Failure categories

| Category | Auto-fixed? | Action |
|----------|------------|--------|
| STRUCTURAL | ✓ (with AI provider) | Re-generated by Claude automatically |
| QUALITY | ✗ | File GitHub issue + `xfail(strict=True)` |
| ENV | ✗ | Network/API unavailable — retry later |

## Output files written automatically

- `apitf/validators/<env>_validator.py` — field contract class
- `tests/test_<env>.py` — pytest test file (or `tests/test_<env>_<resource>.py` in parallel mode)
- `config/environments.yaml` — env block added if absent (includes `probe_path` + `thresholds`)
- `pytest.ini` — marker added if absent
- `test_plans/<env>_test_plan.md` — test plan generated from spec
- `bugs/BUG_REPORT_<env>_<resource>.md` — per-resource bug report (parallel mode)
